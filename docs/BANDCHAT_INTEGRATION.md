# BandChat Website Template — Integration Specification

> This document is for the BandChat development team. It describes how the
> website template works, how it should integrate with BandChat's admin UI,
> and what the end-to-end flow looks like for a non-technical band admin.

## 2. The Template: What It Is

A React + Vite single-page application that:

- Pulls all band data (gigs, members, songs, setlists, media) from BandChat's API via a sync script
- Renders a fully-featured band website with 19 toggleable pages
- Is configured entirely via one file: `site.config.js`
- Uses environment variables for secrets (API keys, credentials)
- Deploys as a static site + serverless functions on Vercel

### site.config.js Structure

```javascript
{
  band: {
    name: "The Rockers",           // Band name
    tagline: "Tokyo Rock Legends", // Shown in hero section
    description: "...",            // Short description
    location: "Tokyo, Japan",      // City, Country
    genre: "Rock Covers",          // Genre label
    founded: 2024,                 // Year
    story: ["paragraph 1", "..."], // About page content
  },
  domain: "therockers.bandchat.com",
  emails: {
    info: "info@therockers.com",
    privacy: "privacy@therockers.com",
  },
  social: [
    { platform: "instagram", url: "https://...", label: "Instagram" },
    { platform: "facebook", url: "https://...", label: "Facebook" },
    { platform: "youtube", url: "https://...", label: "YouTube" },
  ],
  theme: {
    primaryAccent: "#ff3250",   // Main brand color (buttons, accents)
    secondaryAccent: "#ffc800", // Secondary color (highlights)
  },
  images: {
    logo: "/images/logos/logo.png",
    ogImage: "https://therockers.bandchat.com/images/logos/logo.png",
    favicon: "/images/icons/favicon.svg",
    heroImages: ["/images/site/hero-1.webp"],
  },
  features: {
    songs: true,        // Song repertoire page
    archive: true,      // Past gigs archive
    setlists: true,     // Setlists page
    timeline: true,     // Band history timeline
    media: true,        // Photo/video gallery
    stats: true,        // Stats and charts
    blog: false,        // Blog (not yet in BandChat)
    community: false,   // Announcements/polls
    merch: false,       // Merchandise page
    trivia: true,       // Random trivia popups
    songRequests: true, // Song request form
    aiAgents: false,    // AI agents panel (admin only)
  },
  seo: {
    title: "The Rockers | Rock Covers in Tokyo",
    description: "Tokyo's rock covers band...",
  },
  agents: null, // AI agent config (or object with amp/buzz/riff)
}
```

### Environment Variables (set in Vercel)

| Variable | Required | Description |
|----------|----------|-------------|
| `SITE_DOMAIN` | Yes | Band's domain (e.g. `therockers.bandchat.com`) |
| `CONTACT_EMAIL` | Yes | Where contact form emails go |
| `BAND_NAME` | Yes | Band name (for email sender) |
| `VITE_BANDCHAT_URL` | Yes | BandChat API URL |
| `VITE_WORKSPACE_ID` | Yes | Band's BandChat workspace ID |
| `SYNC_BANDCHAT_URL` | Yes | Same as VITE_BANDCHAT_URL |
| `SYNC_EMAIL` | Yes | BandChat login for sync script |
| `SYNC_PASSWORD` | Yes | BandChat password for sync script |
| `SYNC_WORKSPACE_ID` | Yes | Same as VITE_WORKSPACE_ID |
| `RESEND_API_KEY` | For email | Resend API key (shared across all sites) |
| `RESEND_FROM_EMAIL` | For email | Sender email address |
| `ANTHROPIC_API_KEY` | For AI agents | Anthropic API key (shared) |

#### State 2: Configuration form
Shown after clicking "Get Started" or when editing an existing site.

**Section 1: Band Identity**

- Description (textarea, 1-2 sentences)

- Band Story (textarea, supports multiple paragraphs)

- Primary Color (color picker, default #ff3250)

**Section 6: Domain**

## 4. Backend: Deployment Flow

When admin clicks **"Deploy Website"**, BandChat backend should:

### Step 1: Fork the template
```javascript
// GitHub API: fork the template repo into a service account or org
const response = await fetch('https://api.github.com/repos/surista/bandchat-website-template/forks', {
  method: 'POST',
  headers: { Authorization: `token ${GITHUB_TOKEN}` },
  body: JSON.stringify({
    name: `website-${bandSlug}`,      // e.g. "website-the-rockers"
    default_branch_only: true,
  }),
});
```

### Step 2: Write site.config.js
```javascript
// GitHub API: create/update file in the forked repo
const configContent = generateSiteConfig(formData); // JSON → JS export
await fetch(`https://api.github.com/repos/${org}/website-${bandSlug}/contents/site.config.js`, {
  method: 'PUT',
  headers: { Authorization: `token ${GITHUB_TOKEN}` },
  body: JSON.stringify({
    message: 'Configure site for ' + formData.band.name,
    content: Buffer.from(configContent).toString('base64'),
    sha: existingSha, // if updating
  }),
});
```

### Step 3: Upload images to R2
```javascript
// Upload logo and hero images to Cloudflare R2
// Store URLs in config
const logoUrl = await uploadToR2(logoFile, `websites/${bandSlug}/logo.png`);
const heroUrls = await Promise.all(
  heroFiles.map((f, i) => uploadToR2(f, `websites/${bandSlug}/hero-${i}.webp`))
);
```

### Step 4: Create Vercel project
```javascript
// Vercel API: create project linked to the forked repo
const project = await fetch('https://api.vercel.com/v11/projects', {
  method: 'POST',
  headers: { Authorization: `Bearer ${VERCEL_TOKEN}` },
  body: JSON.stringify({
    name: `bandchat-${bandSlug}`,
    gitRepository: {
      type: 'github',
      repo: `${org}/website-${bandSlug}`,
    },
    framework: 'vite',
  }),
});
```

### Step 5: Set environment variables
```javascript
// Vercel API: set env vars for the project
const envVars = [
  { key: 'SITE_DOMAIN', value: `${bandSlug}.bandchat.com` },
  { key: 'CONTACT_EMAIL', value: formData.emails.info },
  { key: 'BAND_NAME', value: formData.band.name },
  { key: 'VITE_BANDCHAT_URL', value: BANDCHAT_API_URL },
  { key: 'VITE_WORKSPACE_ID', value: workspaceId },
  { key: 'SYNC_BANDCHAT_URL', value: BANDCHAT_API_URL },
  { key: 'SYNC_EMAIL', value: serviceAccountEmail },
  { key: 'SYNC_PASSWORD', value: serviceAccountPassword },
  { key: 'SYNC_WORKSPACE_ID', value: workspaceId },
  { key: 'RESEND_API_KEY', value: SHARED_RESEND_KEY },
  { key: 'RESEND_FROM_EMAIL', value: `${formData.band.name} <noreply@bandchat.com>` },
];

for (const env of envVars) {
  await fetch(`https://api.vercel.com/v10/projects/${projectId}/env`, {
    method: 'POST',
    headers: { Authorization: `Bearer ${VERCEL_TOKEN}` },
    body: JSON.stringify({ ...env, type: 'encrypted', target: ['production'] }),
  });
}
```

### Step 6: Add custom domain
```javascript
// Vercel API: add bandname.bandchat.com domain
await fetch(`https://api.vercel.com/v10/projects/${projectId}/domains`, {
  method: 'POST',
  headers: { Authorization: `Bearer ${VERCEL_TOKEN}` },
  body: JSON.stringify({ name: `${bandSlug}.bandchat.com` }),
});
```

### Step 7: Trigger initial data sync
```javascript
// Either:
// a) Call the sync script via a Vercel serverless function
// b) Run sync from BandChat backend and commit the data to the repo
// c) Use a Vercel deploy hook that runs sync as part of the build
```

### Step 8: Trigger deployment
```javascript
// Vercel API: create deployment (or it auto-deploys from the git push)
await fetch('https://api.vercel.com/v13/deployments', {
  method: 'POST',
  headers: { Authorization: `Bearer ${VERCEL_TOKEN}` },
  body: JSON.stringify({
    name: `bandchat-${bandSlug}`,
    gitSource: {
      type: 'github',
      org: org,
      repo: `website-${bandSlug}`,
      ref: 'main',
    },
  }),
});
```

### Step 9: Store deployment info in BandChat DB
```javascript
// Save to workspace record:
workspace.website = {
  repoUrl: `https://github.com/${org}/website-${bandSlug}`,
  siteUrl: `https://${bandSlug}.bandchat.com`,
  vercelProjectId: projectId,
  deployedAt: new Date(),
  config: formData,
};
```

## 5. Data Sync Strategy

The website uses static JSON files synced from BandChat. Two sync strategies:

### Option A: Build-time sync (simplest)
Add to the Vercel build command:
```
node scripts/sync.js && vite build
```
Data is synced every time the site rebuilds. Trigger a rebuild when data changes in BandChat.

### Option B: Webhook-triggered sync
When data changes in BandChat (new gig, updated member, etc.):
1. BandChat calls a Vercel deploy hook URL
2. Vercel rebuilds the site, running sync as part of the build
3. Site updates within ~60 seconds

### Option C: Manual sync button
Admin clicks "Sync Data" in the Website tab:
1. BandChat backend runs the sync script
2. Commits updated JSON to the band's repo
3. Vercel auto-deploys from the commit

**Recommended: Option B** (webhook) for automatic updates, with Option C as a manual fallback.

## 6. DNS Setup for *.bandchat.com

### One-time setup (you do this once):

1. Add a wildcard CNAME record on `bandchat.com`:
   ```
   *.bandchat.com  CNAME  cname.vercel-dns.com
   ```

2. In Vercel, each project gets its subdomain added via API (Step 6 above).

3. Vercel handles SSL automatically for each subdomain.

### Custom domains (optional per band):
If a band wants `therockers.com`:
1. Band buys the domain
2. Band adds CNAME record: `therockers.com → cname.vercel-dns.com`
3. BandChat backend adds the domain to their Vercel project via API
4. Show DNS instructions in the Website tab

## 9. Security Considerations

- **Service account**: Create a dedicated BandChat user for sync operations (not the admin's personal credentials)
- **GitHub token**: Use a fine-grained token scoped to only the template repos
- **Vercel token**: Use a team-level token, not personal
- **Shared API keys**: Resend and Anthropic keys are shared across all band sites via Vercel env vars — scope spending limits
- **CSP headers**: Each band site's `vercel.json` needs its CSP `img-src` updated to include R2 domain and any custom image hosts
